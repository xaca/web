Pseudocódigo

Es un falso lenguaje de programación, se usa para describir en lenguaje natural (español) una serie de pasos que permiten la solución de un problema. Cuando se explica una dirección, una receta, o cualquier procedimiento se esta creando el pseudocódigo de lo que se quiere comunicar.

Pasos para crear un pseudocódigo

  • Enumerar cada paso
  • El primer paso siempre es el inicio y el último es el fin
  • Se debe garantizar que siempre se llegue hasta el fin
  • Los pasos deben tener prioridad
  • Se permite crear saltos entre pasos, se recomienda usar la frase "ir a __" seguido del número del paso
  • Se debe escribir el mayor número de pasos posible

Prueba de escritorio

Una prueba de escritorio es una forma de validar la solución que se ha pensado previamente en el pseudocódigo y en el diagrama de flujo. Se recomienda hacer prueba por cada herramienta, ya que cada vez, se conoce mejor la solución, se puede completar, detectar casos especiales y mejorar el funcionamiento. Además esta prueba permite evitar errores en tiempo de ejecución, estos errores son difíciles de detectar mientra se escribe el programa, pero en la prueba de escritorio, salen a flote con la interacción.

Factorial de un número

En el siguiente ejemplo, se observa como se puede realizar la prueba de escritorio de un diagrama que soluciona la necesidad de calcular el factorial de un número "n". Note que el algoritmo es la solución general, sin embargo en la prueba de escritorio, se prueban dos o tres valores, los más representativos. Además se recomienda probar que pasa con casos especiales, para validar la respuesta del algoritmo a situaciones extremas. Por ejemplo, el factorial es un número que crece rapidamente, por tanto puede desbordar la capacidad de un computador o calculadora muy rápido.

La prueba de escritorio se realiza con lapiz, borrador y un papel. Hay que probar varios pasos, hasta identificar comportamientos correctos o incorrectos del algoritmo. Si se detecta algo por mejorar, se debe ir a el pseudocodigo o al diagrama, ajustar y nuevamente probar más casos, hasta llegar a un estado de tranquilidad para escribir el programa en el lenguaje de programación.
